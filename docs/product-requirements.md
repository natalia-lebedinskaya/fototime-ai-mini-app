# Product Requirements

## Название продукта

Fototime AI Telegram Mini App

## Цель продукта

Создать Telegram Mini App, позволяющий пользователю получить AI-фото на основе загруженного изображения, выбранного участника мероприятия и выбранного стиля обработки.

Приложение должно повторять основной сценарий Telegram-бота FOTOTIME323, но в формате полноценного Mini App с более удобным и визуально привлекательным интерфейсом.

## Бизнес-контекст

В текущем рабочем процессе стили создаются и настраиваются во внешнем сервисе NeuralSolutions.

Администратор:

1. Выбирает или создаёт стили в каталоге стилей.
2. Добавляет нужные стили в мероприятие.
3. Привязывает Telegram-бота к мероприятию.
4. Настраивает доступных участников.
5. Запускает мероприятие.

После этого пользователь в Telegram может выбрать участника, стиль обработки и отправить фото для генерации.

## Целевая аудитория

- гости мероприятий;
- клиенты фотобудки;
- подписчики Telegram-канала;
- пользователи, которым интересно быстро получить стилизованное AI-фото;
- потенциальные клиенты услуги FOTOTIME323.

## Основная ценность продукта

Пользователь получает более удобный и визуально привлекательный интерфейс для выбора участника, стиля и загрузки фото, чем стандартный сценарий через кнопки Telegram-бота.

## Основной пользовательский сценарий

1. Пользователь открывает Telegram Mini App из бота FOTOTIME323.
2. Приложение определяет активное мероприятие.
3. Приложение загружает доступных участников мероприятия.
4. Приложение загружает стили, добавленные в мероприятие.
5. Пользователь выбирает участника.
6. Пользователь выбирает стиль обработки.
7. Пользователь загружает фото.
8. Пользователь запускает генерацию.
9. Пользователь получает результат.
10. Пользователь может создать новое изображение.

## Основные функции

### 1. Открытие Mini App

Пользователь открывает приложение из Telegram-бота.

Ожидаемый результат:

- приложение открывается;
- отображается стартовый экран;
- пользователь видит название продукта и описание сценария;
- приложение готово получить конфигурацию мероприятия.

### 2. Получение конфигурации мероприятия

Приложение должно получить данные активного мероприятия.

Минимальный набор данных:

- eventId;
- botId;
- название мероприятия;
- список доступных участников;
- список доступных стилей;
- настройки языка;
- настройки ориентации результата;
- статус мероприятия.

Ожидаемый результат:

- приложение показывает только те стили, которые добавлены в мероприятие;
- приложение показывает только тех участников, которые включены в мероприятии;
- если мероприятие не найдено, пользователь видит понятное сообщение.

### 3. Выбор участника

Пользователь должен выбрать одного участника из списка, доступного в мероприятии.

Примеры участников:

- Мужчина;
- Женщина;
- Пара;
- Мальчик;
- Девочка;
- Семья.

Ожидаемый результат:

- выбранный участник визуально подсвечивается;
- пользователь может изменить выбранного участника до запуска генерации;
- список участников не должен быть захардкожен на клиенте.

### 4. Выбор стиля

Пользователь выбирает один стиль из списка стилей, добавленных в мероприятие.

Стили должны приходить из backend/API, а не быть захардкожены в frontend.

Пример стиля:

- id;
- name;
- previewUrl;
- participantType;
- price;
- isAvailable.

Ожидаемый результат:

- пользователь видит карточки стилей с превью;
- пользователь видит только стили, подходящие выбранному участнику;
- выбранный стиль визуально подсвечивается;
- пользователь может изменить стиль до запуска генерации.

### 5. Загрузка фото

Пользователь может загрузить изображение.

Поддерживаемые форматы:

- JPG;
- JPEG;
- PNG.

Ограничение размера файла:

- максимум 10 MB.

Ожидаемый результат:

- валидный файл принимается;
- невалидный файл отклоняется;
- пользователь видит понятное сообщение об ошибке.

### 6. Генерация изображения

После выбора участника, стиля и загрузки фото пользователь может запустить генерацию.

Ожидаемый результат:

- запрос отправляется на backend;
- backend передаёт данные в сервис генерации;
- кнопка генерации блокируется на время обработки;
- повторный клик не создаёт дублирующий запрос;
- пользователь видит loading-state.

### 7. Получение результата

После успешной генерации пользователь видит итоговое изображение или mock-result на этапе разработки.

Ожидаемый результат:

- отображается success-state;
- отображается блок результата;
- пользователь может скачать результат;
- пользователь может начать сценарий заново.

## Ограничения

- Генерация недоступна без выбранного участника.
- Генерация недоступна без выбранного стиля.
- Генерация недоступна без загруженного фото.
- Нельзя отправлять несколько запросов генерации одновременно.
- Список стилей не должен быть захардкожен на frontend.
- Список участников не должен быть захардкожен на frontend.
- При ошибке сервиса пользователь должен получить понятное сообщение.
- Технические ошибки не должны отображаться пользователю в сыром виде.
- API-ключи и токены не должны храниться в репозитории.
- API-ключи и токены не должны попадать в frontend-код.

## Нефункциональные требования

### UI

Интерфейс должен быть:

- адаптивным;
- читаемым на мобильных устройствах;
- удобным для использования внутри Telegram Mini App;
- визуально отличающимся от стандартного сценария Telegram-бота;
- похожим на полноценную витрину AI-эффектов;
- ориентированным на wow-effect.

### Производительность

Приложение должно:

- быстро открываться;
- не блокировать интерфейс во время генерации;
- показывать состояние загрузки;
- корректно обрабатывать долгий ответ от сервера;
- не загружать лишние стили, если выбран конкретный участник.

### Надёжность

Приложение должно:

- валидировать файл на клиенте;
- валидировать файл на сервере;
- обрабатывать ошибки API;
- обрабатывать потерю соединения;
- не отправлять дублирующие запросы;
- корректно работать, если список стилей пустой;
- корректно работать, если мероприятие неактивно.

### Безопасность

Приложение должно:

- не хранить секреты в frontend-коде;
- использовать переменные окружения для токенов;
- валидировать входящие данные на backend;
- не доверять данным, пришедшим только с клиента;
- проверять Telegram initData на backend перед выполнением чувствительных операций.

## MVP

В первую версию MVP входит:

- открытие Mini App;
- получение mock-конфигурации мероприятия;
- отображение участников;
- отображение стилей по выбранному участнику;
- выбор участника;
- выбор стиля;
- загрузка фото;
- mock-генерация результата;
- обработка основных негативных кейсов.

## Out of Scope для первой версии

В первую версию не входит:

- реальная интеграция с NeuralSolutions API;
- платёжная логика;
- списание баланса;
- хранение истории генераций;
- полноценная админ-панель;
- управление мероприятиями из Mini App.
